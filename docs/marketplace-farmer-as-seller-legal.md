# Маркетплейс: фермер-като-продавач — юридика и легални данни (Стъпка 1)

Решение 2026-07-12 (owner DNDonchev): farmmarket.bg минава на модел, в който **всеки
фермер е легалният Продавач** — праща със **своя** товарителница, събира **своя** наложен
платеж по **свой** IBAN и се отчита **самостоятелно пред НАП**; платформата взима 10%
комисиона (фактура, след „доставена+платена"). Тази стъпка покрива **юридиката + легалните
данни**; тя не пипа транзакционния слой (split-ът и комисионата вече съществуват спящи).

> ⚠️ Правният текст и счетоводната форма трябва да се потвърдят от **юрист и счетоводител**
> преди go-live. Този документ е техническо задание/структура, НЕ финален правен текст.

## 1. Легални данни на продавача (ГОТОВО в този repo)

Добавена е `farmers.legal` (jsonb, миграция **0092**), огледало на `branding`/`coverCrop`:

| Поле | Смисъл |
|---|---|
| `kind` | `individual` (физ. лице / рег. земеделски производител) · `sole_trader` (ЕТ) · `company` (ЕООД/ООД/АД) |
| `name` | юридическо/фирмено име на продавача |
| `eik` | ЕИК/БУЛСТАТ (ЕТ/фирма) |
| `vatNumber` | ДДС № (ако е по ЗДДС; по избор) |
| `address` | адрес на управление/седалище/кореспонденция |
| `regNo` | рег. № земеделски производител (Наредба 3), за физ. лице |
| `confirmedAt` | кога операторът/фермерът е потвърдил данните (одит) |

- **DTO:** `LegalDto` (`server/.../farmers/dto/legal.dto.ts`), закачен в Create/Update-FarmerDto.
- **Вход:** `FarmerPanel` (client) — секция „Юридически данни · продавач", gated на `multiFarmer`.
- **Публично изложение:** `findPublicBySlug` проектира `legal` явно → storefront-ът може да
  покаже кой е продавачът. За разлика от `commissionRateBps`/`subscriptionFeeStotinki`
  (собственик-only, стриват се), `legal` е **задължителна публична КЗП декларация**.

## 2. Общи условия → посредник (СЛЕДВА, в chaika repo + юрист)

Текстът на Общите условия живее в **отделния storefront repo** (chaika / `farmflow-marketplace`,
farmmarket.bg), НЕ в този backend. Промени за юриста:

- FarmMarket = **„доставчик на онлайн място за търговия / посредник"**, НЕ „Продавач".
- Продавач на всеки продукт = **конкретният фермер** (виж легалните данни от §1).
- Общата кошница създава **отделен договор с всеки продавач**.
- Фермерът отговаря за: продукта, документа за продажба, изпращането, рекламацията.
- FarmMarket отговаря за: сайта, предаването на поръчката, клиентско съдействие.
- Възстановяване на пари: не се обещава от платформата като Продавач (днешният текст твърди
  обратното — това е основното противоречие, което махаме).
- Поверителност + документи за продажба — да се преработят за marketplace модел.

Референции (за юриста): Указания на КЗП за платформи-посредници; Указания на НАП за
пощенски паричен превод и документ за продажба.

## 4. Стъпка 3 — per-farmer Еконт / директен НП (ГОТОВО в този repo)

**Дискавъри:** per-farmer Еконт инфраструктурата ВЕЧЕ съществуваше (courier-per-farmer,
migr 0069-71). Всеки фермер може да свърже СВОЙ Еконт акаунт под
`tenants.settings.delivery.farmers[<id>].econt` (username/passwordEnc/sender/cod); `resolveCreds`
/`loadStored`/`callTenant` са threaded с optional `farmerId`. Еконт сетълва НП към акаунта,
който създава товарителницата → ако waybill-ът е с creds на фермера, парите отиват на
**неговия** регистриран IBAN/пощенски превод автоматично. **Не са нужни IBAN колони.**

Реалните дупки бяха 2 таргетирани поправки (без миграция):

- **(A) `econt.service.createLabel`** — per-farmer order вече ползва creds на СВОЯ фермер
  (не tenant). Преди: само farmer-standalone path подаваше farmerId; operator/admin path
  (`econt.controller` finalize) го изпускаше → НП към tenant акаунт. Сега: резолвира
  `credsFarmerId` от `order.farmerId`, ако този фермер има свързан Еконт; иначе graceful
  fallback към tenant (legacy/tenant-level + неонбординг-нат фермер = без промяна).
- **(B) `consolidation.helpers.groupConsolidationCandidates`** — same-seller-only key
  (`f:<farmerId>|<dest>`). Никога не fold-ва НП на РАЗЛИЧНИ фермери в един master (това
  беше „един събирач" = отхвърленият модел). Един фермер с 2 поръчки към същия клиент
  все още се обединява (един waybill, неговият акаунт).

Тестове: `econt.service.spec` (admin finalize → creds на фермера / fallback към tenant),
`consolidation.helpers.spec` (различни фермери НЕ се обединяват). Всичко зелено; server tsc чист.

⚠️ Остава за go-live: всеки фермер трябва **сам** да свърже своя Еконт акаунт в панела
(вход вече съществува). Без него → fallback към tenant акаунт (НП не отива директно).
