## Data model / key facts

- `settings.delivery` (jsonb): `methods.{ownSlots,pickup,econtOffice,econtAddress}.{enabled,pricing,label,...}`, `econt.mode` (off/manual/auto) + `econt.configured`, `cod.enabled`, `pricing.freeThresholdStotinki`.
- Server helpers: `server/src/modules/orders/delivery-pricing.ts` — `buildPublicMethods`, `buildPublicDelivery`, `econtMode`, `codEnabled`, `econtFallbackFee`, `DELIVERY_DEFAULTS`.
- Profile assembled in `server/src/common/cache/public-cache.service.ts` (`TenantMeta`, Redis-cached per slug) → returned by `server/src/modules/tenants/tenants.service.ts` `findPublicProfileBySlug` (strips internal id/stripeAccountId, derives `stripeEnabled` via injected `StripeService`).
- `delivery_type` enum: `pickup` | `address`(=лична/slots) | `econt`(=office) | `econt_address`(=courier-to-address).
- Admin saves delivery via `client/src/lib/api-client.ts saveDelivery` → busts the tenant Redis cache.
- ⚠️ **Default mismatch bug:** `client/src/lib/delivery-data.ts` `DEFAULT_DELIVERY` (econt 499 / door 690) ≠ server `DELIVERY_DEFAULTS` (350 / 590). Admin save re-persists the hydrated config → materializes the client defaults → silently changes fees. Fix when touching this: align the two, or stop `hydrateDelivery` from baking in pricing the farmer never set.
