# Обобщен приемо-предавателен протокол, протокол по имейл при потвърждаване и готовност на фермерите

**Дата:** 2026-07-21
**Статус:** одобрен дизайн, чака план за изпълнение
**Мостра:** `FarmMarket_obsht_priemno_predavatelen_protokol.docx` (предоставена от клиента)

## Проблем

Днес има само **двустранни** протоколи: един фермер → оператор, или оператор → един
клиент. За курс с осем фермера и двайсет поръчки това са двайсет и осем отделни листа.
Липсва документът, който куриерът реално носи — един обобщен протокол за целия курс.

Освен това протоколът на клиента се ражда чак при предаването. Клиентът няма нищо в
ръцете си между потвърждаването и вратата.

И трето: операторът не вижда кой фермер си е попълнил правните данни и подписа. Разбира
се, че не си ги е попълнил, чак когато протоколът излезе с празни полета.

## Обхват

1. Нов **обобщен протокол** по мостра — секции А (фермери и приет товар), Б (разпределение
   по поръчки), В (приемане от транспортния оператор). Два обхвата: дневен (всички куриери)
   и курierски (един лег). Жив докато е чернова, замразява се при подпис. Редактируем от
   фермер-админ.
2. **Двустранният протокол на клиента се праща по имейл при потвърждаване на поръчката.**
   Единичното потвърждаване чака реалното пращане.
3. **Общ визуален слой** за двата документа — един бранд блок, една типография.
4. **Готовност на фермерите** в секцията с протоколите: правни данни + подпис, с назовано
   какво липсва.

**Извън обхвата:** електронен подпис с квалифициран сертификат; износ към счетоводство;
редакция на подписан документ; автоматично пращане на куриерските протоколи без човек.

## 1. Данни

### 1.1 Нова таблица `consolidated_protocols` (миграция `0112_consolidated_protocols.sql`, journal idx 110)

> ⚠️ Номерът е „следващият свободен към 2026-07-21" (main завършва на `0111` / idx 109).
> Паралелна сесия може да заеме 0112 преди този план да стигне до изпълнение — номерът се
> сверява с `_journal.json` **непосредствено преди** писането на миграцията, не сега.
> Дублиран idx чупи мигратора мълчаливо; това вече се случи веднъж днес.

| колона | тип | бележка |
|---|---|---|
| `id` | uuid pk | `uuid_generate_v4()` |
| `tenant_id` | uuid → `tenants` not null | |
| `scope` | text not null | `'day'` \| `'leg'` |
| `date` | date not null | БГ ден на курса |
| `leg_index` | integer | 0-базиран лег; `null` при `scope='day'` |
| `doc_number` | integer not null | собствена серия, вж. 1.2 |
| `status` | text not null default `'draft'` | `'draft'` \| `'signed'` |
| `meta` | jsonb | ръчните полета от бланката (1.3) |
| `overrides` | jsonb | слой редакции (1.4) |
| `frozen_rows` | jsonb | `null` докато е чернова; попълва се при подпис (1.5) |
| `receiver_signature_png` | text | шифрован, същият `secret.util` както `farmers.signature_png`; вж. 1.8 |
| `signed_at` | timestamptz | |
| `created_at` / `updated_at` | timestamptz | |

Индекси:

- `unique (tenant_id, doc_number)`
- `unique (tenant_id, date, scope, COALESCE(leg_index, -1))` — **`COALESCE` е задължителен.**
  В Postgres NULL-овете са различни помежду си в уникален индекс, така че без него два реда
  с `scope='day'` и `leg_index IS NULL` минават през ограничението и се получават два
  дневни протокола за един ден.

### 1.2 Номерация

Собствена серия с видим префикс: **`ОБ-7`**, докато двустранните остават `7`. Ако двете
теглят от един брояч, „Протокол №7" значи два различни документа — недопустимо пред
контролен орган.

Заделянето използва същия модел като `handover.service.ts:847-862` — advisory lock +
повторна проверка вътре в транзакцията.

### 1.3 `meta` — ръчните полета от бланката

```
{ vehicle?, plate?, driverName?, startPlace?, startTime?, plannedEnd? }
```

Празни по подразбиране. `driverName` се предлага от назначението в
`route_courier_assignments`, но остава редактируемо — колата и водачът се сменят сутринта.

### 1.4 `overrides` — слоят редакции

```
{
  excludedOrderIds: uuid[],        // махнат ред
  extraRows: [{ section: 'A'|'B', ...cells }],   // ръчно добавен ред
  fieldOverrides: { [rowKey]: { batch?, eDoc?, note? } }
}
```

`rowKey` е `f:<farmerId>` за секция А и `o:<orderId>` за секция Б.

### 1.5 Жив докато е чернова, замразен при подпис

**Чернова:** секции А и Б се смятат наживо от текущите поръчки за деня (и лега), после се
налага `overrides`. Късно добавена поръчка влиза сама; ръчното не се губи.

**При подпис:** изчисленият резултат се записва целият в `frozen_rows`, `status='signed'`,
`signed_at` се попълва. Оттам нататък PDF-ът се рендерира **от `frozen_rows`**, не от
поръчките. Това е юридическият запис — какво е предадено в онзи момент.

### 1.6 Нови колони на `orders`

| колона | тип | бележка |
|---|---|---|
| `protocol_email_status` | text | `null` = не е опитвано; иначе `'sent'` \| `'failed'` \| `'bounced'` |
| `protocol_email_at` | timestamptz | час на последния опит |
| `protocol_email_error` | text | причина при `failed` |

`bounced` се попълва от съществуващия Resend webhook, който вече храни `SuppressionService`.

### 1.7 Кой подписва къде

Документът има три подписни места и те идват от три различни източника:

| Място | Подпис | Откъде |
|---|---|---|
| Секция А, по ред на фермера | фермерът | `farmers.signature_png`, отпечатва се готов; празна чертичка, ако липсва |
| Секция В, „приел за транспорт" | куриерът/водачът | `receiver_signature_png` — **хваща се при подписване** на устройството |
| — | | предлага се попълнен от `tenants.operator_signature_png`, когато подписващият е собственикът |

Отделен съхранен подпис на куриер (по подобие на фермерския) **не** се въвежда — куриерът
подписва на място, което е и правно по-честно за „приел за транспорт в час X".

### 1.8 Поправка: `orderIds` за фермерските протоколи

Днес `orderIds` се пише само за клиентския крак; фермерският пише `null`
(`handover.service.ts:876`, `:980`, `:1064`). Заради това клонът в `listForCheck:1142`,
който филтрира по масива, е мъртъв за фермерски протоколи — коментарът на `:1132` твърди,
че покрива многопоръчкови прибирания, а не ги покрива. Обобщеният протокол стъпва точно
на връзката фермер→поръчки, затова се попълва.

## 2. Обхват на документа и кой какво вижда

**Терминология:** „фермер-админ" в тази спецификация значи роля **`admin`** (собственикът на
наемателя / операторът), не роля `farmer`. Роля `farmer` е производителят и няма достъп до
обобщения протокол — той вижда само собствения си двустранен.

| Обхват | За кого | Съдържание |
|---|---|---|
| `scope='day'` | фермер-админ (`admin`) / офис | всички фермери и всички поръчки за деня, всички куриери |
| `scope='leg'` | по един на всеки активен куриер | само неговите фермери и поръчки |

„Активен куриер днес" **не** се измисля наново — това е точно съдържанието на
`route_courier_assignments` за датата.

**Куриерът на пътя вижда единствено своя `scope='leg'`.** Дневният съдържа чужди поръчки;
да го покаже на полицай значи да разкрие поръчки, които не носи. Това е същото правило,
което вече е наложено на `GET /handover/check` (scope през `resolveMyLeg` + `getRoute(…,'all')`).

Ако фермер-админът е и куриер за деня, получава и двата.

## 3. PDF

### 3.1 Общ слой `pdf-kit.ts`

Изнасят се: шрифтът, размерите на листа, `wrap()`, и **новите** `drawDocumentHeader()` /
`drawDocumentFooter()`. Двата рендера викат едни и същи — това е механизмът за общия вид,
а не копиране на стилове. Бранд блокът носи марката, заглавието, номера и датата.

### 3.2 Табличен примитив

`drawTable({ columns, rows })` — пренасяне във всяка клетка, височина на реда според
най-високата клетка, **повтарящ се заглавен ред на всяка нова страница**. Днес няма нищо
такова: рендерът е курсор `y` и две текстови функции, без клетки и колони.

### 3.3 Страниране

Цикъл с прелив. Прилага се и на **съществуващия** двустранен протокол — там `addPage` се
вика точно веднъж (`handover-pdf.ts:158`) и дълго съдържание тихо пада извън листа. Това е
бъг и днес; файлът се пипа така или иначе.

### 3.4 Шрифт

Добавя се `DejaVuSans-Bold.ttf` до текущия. Днес удебеляването е рисуване два пъти с 0.4pt
отместване (`:179`, `:238`) — за таблица със заглавни редове изглежда мърляво. Лицензът е
същият, вече е в `assets/fonts/LICENSE.txt`.

### 3.5 Ориентация

Хоризонтален A4 за обобщения, вертикален остава за двустранния. Колоната „ПРОДУКТИ И
КОЛИЧЕСТВА" в секция Б е дълга и на вертикален лист става на каша. `PAGE_W`/`PAGE_H` се
четат от `handover-pdf.spec.ts`, затова се **параметризират**, а не се променят.

Новият рендер е отделен файл (`consolidated-pdf.ts`), не преработка на стария.

### 3.6 Отклонение от бланката: подписите в секция А

Бланката има ред с десет празни чертички. При нас фермер с качен подпис се печата
**подписан**; празна чертичка остава само за тези без. Иначе десет реда фермери значат
десет ръчни подписа на вратата всяка сутрин — а точно това трябваше да отпадне.

### 3.7 Текстът за поверителност в секция Б

Изричният ред от бланката се запазва дословно: име, телефон и точен адрес на клиента
остават само в защитения маршрутен списък. Документът показва номер на поръчка, клиентски
код и град/зона. Това съвпада с вече наложеното поведение на екрана „Проверка".

## 4. Имейл

### 4.1 Прикачени файлове

`SendMailOptions` днес няма `attachments`, а `sendMail` слага целия обект в Redis като JSON.
Затова в задачата отива **описание, не съдържание**:

```ts
attachments?: { kind: 'handover-protocol'; protocolId: string }[]
```

PDF-ът се материализира в момента на пращането, вътре в `deliver()`. Задачата остава
няколко байта; ретраят рендерира наново, вместо да препраща байтове отпреди часове.
`writePreview()` в dev също трябва да отрази прикачения файл, а не да го изпусне мълчаливо.

### 4.2 Ред на операциите при потвърждаване

Не се използва компенсиращ откат („потвърди, прати, при провал върни статуса") — той
оставя точно дупката, която трябва да няма: `confirmed` без имейл, ако и откатът гръмне.

Редът е обърнат:

1. Подготвя се протоколът и се рендерира PDF-ът.
2. `await` реалното пращане — **без отворена транзакция**, за да не се държи ред заключен
   през SMTP.
3. Чак тогава статусът става `confirmed`.

Така невъзможният случай е непоносимият. Остава поносимият: имейлът е тръгнал, а записът
е гръмнал → поръчката стои непотвърдена, вижда се, и повторният опит е безопасен (пази се
`protocol_email_status='sent'`).

**Спънка:** `handover.service.ts:30` смята за готови само поръчки в `confirmed|preparing`.
Рендерирането преди флипа минава по изричен `orderId`, не през дневния филтър по статус.

### 4.3 Поведение по пътища

| Път | Поведение |
|---|---|
| `PATCH /orders/:id/status` (човек пред екрана) | чака имейла, проваля се шумно |
| `PATCH /orders/confirm-pending` (масово) | опашка, отговорът връща „X от Y не тръгнаха" |
| Stripe webhook | опашка — Stripe чака бърз 200 |

И трите пишат в `protocol_email_status`. На поръчката има бутон „прати пак".

**„Изпратен" значи „приет от пощенския сървър", не „получен".** UI-ят не твърди, че
клиентът го е получил; bounce-ите идват по webhook и вдигат `bounced`.

**Прикаченият протокол е НЕподписан** — подписването става при предаването, часове по-късно.
Върху PDF-а се отбелязва явно, че е предварителен, иначе се размахва неподписан лист като
доказателство.

### 4.4 Куриерските протоколи

Пращат се с **бутон** от секцията с протоколите („Прати на куриерите"), не автоматично.
Маршрутът се пренарежда до последно; автоматичното пращане значи трима куриери с по четири
версии в пощата. Бутонът показва списъка с получатели преди да тръгне.

## 5. Готовност на фермерите

### 5.1 Проверката

| Проверка | Източник |
|---|---|
| Вид лице избран | `farmers.legal.kind` |
| Име | `farmers.legal.name` |
| Идентификатор **според вида** | `eik` за фирма/ЕТ, `regNo` за физическо лице |
| Адрес | `farmers.legal.address` |
| Подпис | `farmers.signature_png` (наличие; съдържанието не се дешифрира за проверката) |

Идентификаторът се проверява според вида, не „някой от двата". Физическо лице с попълнен
ЕИК не е готово, а невярно попълнено — това е бъгът, затворен с `buildLegalPayload`
(`client/src/lib/legal-identity.ts`). Логиката се преизползва, не се преписва.

### 5.2 Екранът

Нов блок „Готовност на фермерите" в секцията с протоколите. По ред на фермер, с **назовано
какво липсва** („няма подпис", „няма адрес"), непълните най-отгоре. Не процент — процентът
не казва кое да се оправи.

До всеки непълен ред двата съществуващи пътя: „Попълни вместо него" (операторът въвежда и
хваща подпис) и „Прати покана" (имейл с връзка фермерът да си попълни сам, през вече
съществуващото `/farmers/me` самообслужване).

### 5.3 Не блокира

Липсващите данни дават предупреждение преди печат — „3 фермера без данни, протоколите им
излизат с празни полета" — но не спират предаването. Стоката тръгва в 6 сутринта; софтуер,
който спира деня заради празно поле, се заобикаля, а не се поправя.

### 5.4 Дребни поправки на списъка

Днес редът показва само `[Вид, Страна, Статус, Действия]`. Добавят се **номерът на
протокола** (списък с правни документи без номера не се сверява с папката) и **броят
редове/поръчки**.

## 6. Сигурност и поверителност

- `scope='day'` е достъпен само за роля `admin`. Куриер получава единствено своя лег, през
  същия механизъм като `/handover/check` (`resolveMyLeg` + филтър по `orderId` **и** по
  масива `orderIds`).
- Подписите остават шифровани в покой (AES-256-GCM, `secret.util`) и не влизат в CRUD
  отговори — типът `FarmerRow` вече изключва колоната.
- Секция Б не съдържа име, телефон и точен адрес на клиента.
- Новите маршрути минават през `TenantRolesGuard`, който отказва по подразбиране към
  `admin`; `@Roles` се добавя явно там, където куриер трябва да достигне.

## 7. Тестове

| Област | Какво се проверява |
|---|---|
| Номерация | едновременно създаване не дава дублиран `doc_number` (advisory lock) |
| Уникалност | два реда `scope='day'` за един ден се отказват — тестът е върху `COALESCE` индекса |
| Жив изглед | късно добавена поръчка влиза; `excludedOrderIds` я маха; `extraRows` оцелява |
| Замразяване | след подпис промяна в поръчките **не** променя PDF-а |
| Обхват | куриер А не вижда поръчки на куриер Б; без назначение → празно, не цял наемател |
| Таблица | пренасяне, височина на реда, повтарящ се заглавен ред на страница 2 |
| Имейл ред | провал на пращането оставя поръчката непотвърдена |
| Имейл идемпотентност | повторно потвърждаване не праща втори път |
| Опашка | задачата съдържа описание, не байтове |
| Готовност | физическо лице с ЕИК е „непълно"; фирма с `regNo` — също |

Тест за скоуп се пише със **записващ mock**, който проверява реалния `WHERE`, а не само
върнатата стойност — зелен тест, сляп за собствения си бъг, вече се е случвал в този модул.

## 8. Ред на изпълнение

Обхватът покрива три доставяеми, които са независими една от друга и **не бива да се
пишат наведнъж**. Планът ги сече на фази, като общата основа е първа:

| Фаза | Съдържание | Зависи от |
|---|---|---|
| 0 | `pdf-kit.ts` (общ бранд блок, bold шрифт, страниране, табличен примитив) + страниране на съществуващия двустранен протокол | — |
| 1 | Обобщен протокол: таблица, миграция, жив изглед + слой редакции, замразяване, рендер, екран за редакция | фаза 0 |
| 2 | Протокол по имейл при потвърждаване: прикачени файлове, обърнат ред на операциите, `protocol_email_status`, трите пътя | фаза 0 |
| 3 | Готовност на фермерите + дребните поправки на списъка + `orderIds` за фермерските протоколи | — |

Фаза 0 е обща, защото изискването „да изглеждат наши" се изпълнява именно там; ако фази 1
и 2 тръгнат паралелно преди нея, двата документа се разминават визуално и после се
помиряват на ръка.

## 9. Известни компромиси

1. **Хоризонтален лист до вертикален** в един пакет PDF-и изглежда смесено. Приема се —
   четимостта на секция Б тежи повече.
2. **Блокиращото пращане добавя латентност** на единичното потвърждаване (порядък 1–3s).
   Съзнателно: това е цената на гаранцията, а масовият път не я плаща.
3. **`frozen_rows` дублира данни**, които иначе се смятат. Съзнателно: подписан документ,
   който се променя след подписа, не струва нищо.
