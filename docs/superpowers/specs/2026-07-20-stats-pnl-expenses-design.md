# Статистика: приходи, ръчни разходи и печалба по куриер

**Дата:** 2026-07-20
**Статус:** одобрен дизайн, чака план за изпълнение

## Проблем

Страницата „Статистика" показва само оборот на стоката. Собственикът няма представа
какви са **неговите** приходи (доставка + информационна комисионна), какви разходи прави
(гориво преди всичко) и кой куриер колко му носи. Няма никакъв модел за разходи в кода.

## Обхват

1. Ръчно въвеждане на разходи (дата, сума, категория, куриер по избор, бележка).
2. Информационна комисионна: един процент за целия магазин, настройван от собственика.
3. Нова секция в Статистика: приходи, разходи, печалба + разбивка по куриер.

Извън обхвата: повтарящи се/абонаментни разходи, разходи по фермер, автоматична
комисионна от `commission_entries` (vendor-finance модулът остава спящ), износ към
счетоводство.

## Данни

### Нова таблица `manual_expenses` (миграция `0111_manual_expenses.sql`, ръчно написана)

| колона | тип | бележка |
|---|---|---|
| `id` | uuid pk | `uuid_generate_v4()` |
| `tenant_id` | uuid → `tenants` not null | on delete cascade |
| `date` | date not null | БГ ден на разхода |
| `amount_stotinki` | integer not null | > 0, под `int4` таван |
| `category` | text not null | `fuel` \| `packaging` \| `salary` \| `fees` \| `other` |
| `courier_account_id` | uuid → `users` | NULL = общ разход, не се води на куриер; `on delete set null` |
| `note` | text | |
| `created_at` | timestamptz default now() | |
| `created_by_id` | uuid → `users` | одит кой е въвел |

Индекси: `(tenant_id, date)` за периодните заявки, `(tenant_id, courier_account_id)` за
разбивката по куриер.

Всички суми са `integer` в стотинки — както навсякъде другаде в кода. Без плаваща запетая.

Миграцията се пише на ръка и се добавя в `meta/_journal.json` без дупка в индексите —
пропуск в номерацията чупи мигратора мълчаливо.

### Настройка

`tenants.settings.stats.infoCommissionBps` — базисни точки (1000 = 10%). Записва се през
`jsonbDeepMerge` (`server/src/common/db/jsonb.ts`), не с read-modify-write: паралелен запис
в `settings` вече е изтривал чужди ключове.

## Сметките

Периодът идва от съществуващия филтър на страницата (`range` / `from` / `to`).

**База на деня: `orders.deliveredAt` (БГ ден).** Куриер печели само когато е доставил;
непоръчаните/недоставени поръчки не участват. Тази база е независима от превключвателя на
`GET /stats/turnover` и не го променя.

За всяка доставена поръчка в периода:

- `itemsStotinki` = `sum(order_items.quantity * order_items.price_stotinki)`
- `deliveryStotinki` = `max(0, orders.total_stotinki − itemsStotinki)` (същата формула,
  която `routing.service.ts` вече ползва за „доставка" на лента)
- `commissionStotinki` = `round(itemsStotinki * infoCommissionBps / 10000)`

Общо:

```
Приходи  = Σ deliveryStotinki + Σ commissionStotinki
Разходи  = Σ manual_expenses.amount_stotinki в периода
Печалба  = Приходи − Разходи
```

Оборотът на стоката (`Σ itemsStotinki`) се показва отделно като контекст и **не** влиза в
„приходи".

### По куриер

Привързване: `orders.deliveredAt` (БГ ден) + `orders.courier_index` →
`route_courier_assignments (tenant_id, date, leg_index)` → `account_id`.

- приход на куриер = доставка + комисионна на неговите доставени поръчки
- разходи на куриер = `manual_expenses` с неговия `courier_account_id`
- печалба на куриер = разлика

Два специални реда:

- **„Общи"** — разходите без куриер. Показват се отделно и **не се разпределят**
  пропорционално между куриерите (разпределението би направило печалбата на куриер лъжлива),
  но влизат в общата печалба.
- **„Неразпределени"** — доставени поръчки, за чийто ден+лента няма ред в
  `route_courier_assignments` (никой не е назначен), както и поръчки с `courier_index` NULL.

Групирането е **по човек (акаунт)**, не по лента: ако Иван кара лента 1 в понеделник и лента 2
във вторник, всичко е на Иван.

## API

Всичко в `stats` модула, `@Roles('admin')` — фермер под-акаунт и шофьор получават 403.

```
GET    /stats/pnl?range|from|to      приходи/разходи/печалба + разбивка по куриер
GET    /stats/expenses?from&to       списък разходи за период
POST   /stats/expenses               нов разход
PATCH  /stats/expenses/:id           редакция
DELETE /stats/expenses/:id           изтриване
PATCH  /stats/commission { bps }     информационен процент
```

Отговор на `/stats/pnl`:

```jsonc
{
  "commissionBps": 1000,
  "goodsTurnoverStotinki": 0,
  "revenue": { "deliveryStotinki": 0, "commissionStotinki": 0, "totalStotinki": 0 },
  "expenses": { "totalStotinki": 0, "byCategory": [{ "category": "fuel", "amountStotinki": 0 }] },
  "profitStotinki": 0,
  "couriers": [{
    "accountId": "…", "name": "Иван",
    "deliveryStotinki": 0, "commissionStotinki": 0,
    "revenueStotinki": 0, "expenseStotinki": 0, "profitStotinki": 0
  }],
  "unassigned": { "deliveryStotinki": 0, "commissionStotinki": 0, "revenueStotinki": 0 },
  "generalExpensesStotinki": 0
}
```

Правила:

- Всички заявки са tenant-scoped. `PATCH`/`DELETE` по `:id` носят `tenant_id` в `WHERE`, не
  само `id` — иначе се пише през граница на наемател.
- Валидация на DTO: `date` — `@Matches(/^\d{4}-\d{2}-\d{2}$/)`; `amountStotinki` —
  `@IsInt() @Min(1) @Max(2_000_000_000)`; `category` — `@IsIn([...])`; `note` — `@IsOptional()`
  + `@Transform` за `'' → undefined`; `courierAccountId` — `@IsOptional() @IsUUID()` и трябва
  да е `role='driver'`/`admin` акаунт на същия наемател.
- `/stats/pnl` се кешира като останалите stats заявки; всеки запис на разход или на процента
  инвалидира ключа за наемателя.
- Празен период връща нули, не `null`.

## UI

Нова секция „Приходи и разходи" в `/stats`, под съществуващите блокове, ползва същия филтър
за период:

1. Три плочки: **Приходи** / **Разходи** / **Печалба** (печалбата е червена при минус).
2. Таблица по куриер + редовете „Общи" и „Неразпределени".
3. Списък разходи за периода + бутон „Добави разход" → диалог (дата, сума, категория, куриер
   по избор, бележка); редакция и изтриване на ред.
4. Поле за информационния процент със „Запази".
5. Мобилно: под 640px таблицата по куриер се сменя с карти. Диалогът се проверява на 375px —
   ширината, на която реално се работи.

Секцията се вижда само при `role === 'admin'`; за фермер под-акаунт изобщо не се рендира
(сървърът така или иначе връща 403 — UI-ят не е защитата).

## Тестове

Backend:

- комисионна: закръгляне на bps, нулев процент → 0
- доставка = `max(0, total − items)`, никога отрицателна
- привързване куриер↔поръчка през `route_courier_assignments`; ден без назначение →
  „Неразпределени"; един човек на различни ленти в различни дни се сумира на него
- общите разходи не се разпределят по куриери, но влизат в общата печалба
- tenant scope: разход на друг наемател не се чете, не се редактира и не се трие
- роля: `farmer` и `driver` → 403
- празен период → нули

Frontend: чиста логика (форматиране, сборове по категория) като `.test.ts` под vitest —
клиентският сет е node-only, няма jsdom/RTL.

## Рискове

- **`deliveredAt` липсва на стари поръчки** — те просто не влизат в периода. Приемливо:
  разбивката по куриер и без това има смисъл само след въвеждането на маршрутите.
- **Смяна на процента преминава ретроактивно през историята** — статистиката е информационна,
  не счетоводна; описано е в UI-я с един ред пояснение.
- **Разход, привързан към изтрит акаунт** — `on delete set null` го превръща в общ разход,
  вместо да го изгуби.
